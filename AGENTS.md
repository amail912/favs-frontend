# AGENTS.md

## What is this repo?
FAVS is a PureScript/Halogen frontend for daily-life task management, centered on a rich calendar. The backend lives at `/home/dev/workspace/foucl2`, but coding agents must not edit that repo.

## Hard Constraints
- Never touch `/home/dev/workspace/foucl2`. If backend work is needed, ask the user.
- Never push to the remote repo.
- Use TDD by default. Add or update focused unit tests and E2E tests for behavior changes.
- Before committing: run all tests, then lint and format, and leave no compiler or linter warnings unaddressed.
- `static/index.js` is generated by the build; do not hand-edit it.

## Stack And Commands
- PureScript + Halogen UI, built/tested with Spago. Package set: `psc-0.15.4-20220805` in `spago.yaml`.
- Node.js + Express dev server/proxy via `http-proxy-middleware`.
- Playwright E2E tests, Bootstrap-based static assets in `static/`.
- `npm run build`: bundles PureScript to `static/index.js`.
- `npm test`: runs `spago test`.
- `npm run lint`: runs `purescript-linter -c purs-lint.json`.
- `npm run format`: runs `purs-tidy format-in-place "src/**/*.purs" "test/**/*.purs"`.
- `npm run start-dev`: serves `http://localhost:1234` and proxies `/api` to `http://localhost:8081`.
- `npm run start-dev-proxy-prod`: same static server, proxy target from `dev/proxy-server-prod.js`.
- `npm run e2e`, `e2e:headed`, `e2e:ui`: build then run Playwright.
- `npm run e2e:install`: install Playwright Chromium.
- `npm run pw:open-session`: open a development Playwright browser session.

## Architecture Map
- Entry point: `src/Main.purs` mounts `Pages.App.component`.
- Root/routing/auth shell: `src/Pages/App.purs`.
  - Browser routes include `/notes`, `/checklists`, `/calendar`, `/admin`, `/signup`, `/signin`.
  - Auth profile is loaded from `GET /api/auth/profile`.
  - `/admin` is guarded by the `admin` role; admin navigation is only visible to admins.
  - Calendar route state supports query params such as `day` and `item`.
- Main pages:
  - `src/Pages/Notes.purs`
  - `src/Pages/Checklists.purs`
  - `src/Pages/Calendar.purs`
  - `src/Pages/Admin.purs`
- Domain and feature modules:
  - `src/Domain/Notes.purs`, `src/Domain/Checklists.purs`
  - `src/Calendar/Recurrence.purs`
  - `src/Calendar/ExImport/*` for CSV/ICS import/export
  - `src/Notifications/LateItems.purs` for late-item derivation and quick-complete logic
- API modules:
  - `src/Api/*`
  - Notes/checklists contracts live in `src/Api/*Contract.purs`.
  - Calendar, trip sharing, and admin payload encoders/decoders live near the relevant API modules.
- UI helpers:
  - `src/Ui/*` contains modals, modal-history behavior, auth session forms, date/time picker, error handling, focus/local-storage/vibration FFI, and CSS helpers.

## Current Flows And API
- Notes/checklists load through `Api.Notes` and `Api.Checklists`: `/api/note` and `/api/checklist` with create/update/delete endpoints on the same resource family.
- Calendar items use `/api/v1/calendar-items`. Creation is `POST`; update currently also uses `POST` to the same collection endpoint; deletion uses `/api/v1/calendar-items/:id`.
- Calendar supports task items and trip items, day/week/month/list views, drag-and-drop, recurrence, contextual create defaults, deletion confirmation, CSV/ICS import/export, and modal browser-history handling.
- Trip sharing and presence use:
  - `GET /api/v1/trip-places`
  - `GET/POST/DELETE /api/v1/trip-sharing/shares`
  - `GET/POST/DELETE /api/v1/trip-sharing/subscriptions`
  - `GET /api/v1/trip-sharing/period-trips?start=...&end=...`
- Shared presence is derived client-side from period trips and displayed in the calendar day view rail/inspection UI. Presence cue preferences are stored client-side.
- Late-item reminders are derived from calendar task items whose end time is past and whose status is not `Done` or `Canceled`. Quick-complete updates status and actual duration through the calendar item update API.
- Admin uses:
  - `GET /api/v1/admin/pending-signups`
  - `POST /api/v1/admin/pending-signups/approve`
  - `DELETE /api/v1/admin/pending-signups/:username`
  - `GET /api/v1/admin/users`
  - `DELETE /api/v1/admin/users/:username`
- Dev proxy: `dev/proxy-server.js` serves `static/`, supports history fallback to `index.html`, and proxies `/api` to the backend. Playwright reuses `http://localhost:1234` if already running.

## Coding Conventions
- Prefer typed domain modeling and compiler feedback over runtime conventions.
- Keep `Api.*` modules technical: no business decisions. Callers decide payload semantics; `EncodeJson`/`DecodeJson` instances live near API contract/types.
- Logic and views for each domain stay in that domain/page module. Avoid centralizing domain behavior in generic helpers.
- Shared helpers must be technical only.
- Domain `render*` functions emit their own domain actions. Page-level callers map those actions into page actions through local helper functions with matching names.
- Domain rendering imports should be qualified with short, meaningful aliases.
- When a module needs to emit multi-domain actions, expose a `*Action` type and let callers map it.
- Prefer one-level destructuring in function signatures and avoid aliases that do not clarify intent.
- Avoid useless suffixes in lens names.
- Avoid `unsafeCoerce` in event handlers; prefer typed conversions such as `fromEvent` when available.

- E2E auth/session helpers live in `e2e/support/` and `e2e/fixtures/`.
- Documentation and backlog live under `docs/`.
- Feature stories live under `docs/features/` and move through `to-refine` -> `ready` -> `done`/`canceled`/`postponed`.
- When finishing a documented story, move it to the appropriate lifecycle folder and record important decisions in the story.
